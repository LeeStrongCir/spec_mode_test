---
description: 通过处理并执行 tasks.md 中定义的所有测试执行任务来完成测试计划。
---

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你 **MUST** 考虑用户输入（如果不为空）。

## 概要

1. 从仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` 并解析 `FEATURE_DIR` 和 `AVAILABLE_DOCS` 列表。所有路径必须是 absolute paths。对于参数中的单引号（如 "I'm Groot"），使用转义语法：`'I'\''m Groot'`（或尽可能使用双引号：`"I'm Groot"`）。

2. **检查检查清单状态**（如果 `FEATURE_DIR/checklists/` 存在）：
   - 扫描 `checklists/` 目录中的所有检查清单文件
   - 对每个检查清单，统计：
     - 总项数：匹配 `- [ ]` 或 `- [X]` 或 `- [x]` 的所有行
     - 已完成项：匹配 `- [X]` 或 `- [x]` 的行
     - 未完成项：匹配 `- [ ]` 的行
   - 创建状态表：

     ```text
     | Checklist | Total | Completed | Incomplete | Status |
     |-----------|-------|-----------|------------|--------|
     | ux.md     | 12    | 12        | 0          | ✓ PASS |
     | test.md   | 8     | 5         | 3          | ✗ FAIL |
     | security.md | 6   | 6         | 0          | ✓ PASS |
     ```

   - 计算总体状态：
     - **PASS**：所有检查清单的未完成项数为 0
     - **FAIL**：一个或多个检查清单存在未完成项

   - **如果有检查清单未完成**：
     - 显示包含未完成项计数的表格
     - **STOP** 并询问："某些检查清单未完成。你是否仍要继续测试执行？(yes/no)"
     - 等待用户回复后再继续
     - 如果用户回复 "no"、"wait" 或 "stop"，中止执行
     - 如果用户回复 "yes"、"proceed" 或 "continue"，进入第 3 步

   - **如果所有检查清单均已完成**：
     - 显示全部通过的状态表
     - 自动进入第 3 步

3. 加载并分析测试执行上下文：
   - **必需**：读取 `tasks.md` 获取完整的任务列表和执行计划
   - **必需**：读取 `plan.md` 获取测试策略、工具链和文件结构
   - **如果存在**：读取 `research.md` 获取技术决策和约束
   - **如果存在**：读取 `spec.md` 获取测试场景与验收标准

4. **项目设置验证**：
   - **必需**：检查测试环境是否就绪（Docker / K8s / 虚拟机）
   - **必需**：验证测试工具和依赖已正确安装
   - **必需**：基于实际项目配置创建/验证 ignore 文件
   - **必需**：确认测试数据集已加载且可用

5. 解析 `tasks.md` 结构并提取：
   - **任务阶段**：Phase 1 测试环境准备 → Phase 2 测试用例编写 → Phase 3 测试自动化代码编写 → Phase 4 测试执行
   - **执行模式**：
     - Phase 1～3 为串行执行（阶段内按顺序完成，不得跳过或并行）
     - Phase 4 中每个自动化用例为**并行执行**（标记 `[P]` 的任务可同时运行）
   - **任务详情**：ID, 描述, 文件路径, 标签（`[ENV]`, `[CASE-SC-01]`, `[AUTO-SC-01]`, `[EVT-SC-01]`）, 并行标记 `[P]`

6. 执行 Phase 1: 测试环境准备（串行）：
   - 搭建测试环境（Docker / 集群 / 虚拟机等）
   - 部署 Mock 服务与测试替身
   - 注入测试数据与 fixtures
   - 验证环境就绪后方可进入 Phase 2

7. 执行 Phase 2: 测试用例编写（串行，按场景顺序）：
   - 为每个测试场景编写手工测试步骤
   - 每个步骤包含：前置条件、操作步骤、预期结果
   - 确认每个用例可被测试执行者理解和执行

8. 执行 Phase 3: 测试自动化代码编写（串行，按场景顺序）：
   - 将 Phase 2 中的手工用例编码为自动化测试脚本
   - 同场景内的多个自动化用例可标注 `[P]` 并行编写
   - 确保每个自动化脚本可直接被 CI/CD 触发

9. 执行 Phase 4: 测试执行（并行）：
   - **每个自动化用例彼此独立可并行执行**
   - 识别所有标记 `[P]` 的任务并**同时触发**
   - 如果任何并行任务 `[P]` 失败，继续执行其余成功的任务，在完成后统一报告失败任务
   - 每个用例执行结果必须记录于测试报告中
   - 失败用例必须产出测试事件报告（Test Incident Report）
   - 所有用例执行完成后执行回归验证

10. 进度跟踪与错误处理：
    - 每完成一个任务后报告进度
    - Phase 1～3 中任何串行任务失败 → **立即中止**，阻塞后续阶段
    - Phase 4 中任何并行任务失败 → **继续执行其余任务**，完成后统一报告
    - 提供清晰的错误消息及调试上下文
    - 如果无法继续执行，建议下一步操作
    - **重要**：对于已完成的任务，确保在 tasks 文件中标记为 `[X]`。

11. 完成验证：
    - 验证所有必需任务已完成
    - 检查执行的测试用例是否与原始规格匹配
    - 验证测试通过且覆盖率满足要求
    - 确认执行遵循测试计划
    - 报告最终状态并提供完成工作摘要

注：此命令假设 `tasks.md` 中存在完整的任务分解。如果任务不完整或缺失，建议先运行 `/speckit-test.tasks` 重新生成任务列表。
