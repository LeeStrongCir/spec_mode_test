---
description: 通过处理并执行 tasks.md 中定义的所有测试执行任务来完成测试计划。
---

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你 **必须** 考虑用户输入（如果不为空）。

## 概要

1. 从仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` 并解析 `FEATURE_DIR` 和 `AVAILABLE_DOCS` 列表。所有路径必须是是绝对路径。对于参数中的单引号（如 "I'm Groot"），使用转义语法：`'I'\''m Groot'`（或尽可能使用双引号：`"I'm Groot"`）。

2. **检查检查清单状态**（如果 `FEATURE_DIR/checklists/` 存在）：
   - 扫描 `checklists/` 目录中的所有检查清单文件
   - 对每个检查清单，统计：
     - 总项数：匹配 `- [ ]` 或 `- [X]` 或 `- [x]` 的所有行
     - 已完成项：匹配 `- [X]` 或 `- [x]` 的行
     - 未完成项：匹配 `- [ ]` 的行
   - 创建状态表：

     ```text
     | 清单 | 总数 | 已完成 | 未完成 | 状态 |
     |-----------|-------|-----------|------------|--------|
     | ux.md     | 12    | 12        | 0          | ✓ 通过 |
     | test.md   | 8     | 5         | 3          | ✗ 失败 |
     | security.md | 6   | 6         | 0          | ✓ 通过 |
     ```

   - 计算总体状态：
     - **通过**：所有检查清单的未完成项数为 0
     - **失败**：一个或多个检查清单存在未完成项

   - **如果有检查清单未完成**：
     - 显示包含未完成项计数的表格
     - **停止** 并询问："某些检查清单未完成。你是否仍要继续测试执行？(yes/no)"
     - 等待用户回复后再继续
     - 如果用户回复 "no"、"wait" 或 "stop"，中止执行
     - 如果用户回复 "yes"、"proceed" 或 "continue"，进入第 3 步

   - **如果所有检查清单均已完成**：
     - 显示全部通过的状态表
     - 自动进入第 3 步

3. 加载并分析测试执行上下文：
   - **必需**：读取 `tasks.md` 获取完整的任务列表和执行计划
   - **必需**：读取 `plan.md` 获取测试策略、工具链和文件结构
   - **如果存在**：读取 `research.md` 获取技术决策和约束
   - **如果存在**：读取 `spec.md` 获取测试场景与验收标准
   - **如果存在**：读取 `data-model.md` 获取实体数据结构和关系
   - **如果存在**：读取 `contracts/` 获取 API 定义
   - **如果存在**：读取 `data-testid.md` 获取页面可交互组件 `data-testid` 标识
   - **如果存在**：读取 `.specify/memory/constitution.md` 获取制品信息与部署要求，以及治理约束
   - **如果存在**：读取 `quickstart.md` 获取测试环启动流程

4. 解析 `tasks.md` 结构并提取：
   - **任务阶段**：阶段1：测试用例编写 → 阶段2：测试自动化代码编写 → 阶段3：测试环境准备 → 阶段4：测试执行
   - **执行模式**：阶段间**严格串行**，但阶段1、2、4**内部任务可并行**
   - **任务详情**：ID, 描述, 文件路径, 标签（`[ENV]`, `[CASE-SC-01]`, `[AUTO-SC-01]`, `[EVT-SC-01]`）, 并行标记 `[P]`
   - **执行流程**：顺序和依赖要求

5. 按照任务计划执行实现：
   - **逐阶段执行**：完成每个阶段后再进入下一阶段
   - **遵循依赖**：按顺序运行顺序任务，并行任务 [P] 可以一起运行  
   - **文件协调**：影响同一文件的任务必须按顺序运行
   - **验证检查点**：在继续之前验证每个阶段的完成情况

6. 进度跟踪与错误处理：
   - 每完成一个任务后报告进度
   - 阶段 1～3 中任何任务失败 → **立即中止**，阻塞后续阶段
   - 阶段 4 中任何并行任务失败 → **继续执行其余任务**，完成后统一报告
   - 提供清晰的错误消息及调试上下文
   - 如果无法继续执行，建议下一步操作
   - **重要**：对于已完成的任务，确保在 tasks 文件中标记为 `[X]`。

7. 完成验证：
   - 验证所有必需任务已完成
   - 检查执行的测试用例是否与原始规格匹配
   - 验证测试通过且覆盖率满足要求
   - 确认执行遵循测试计划
   - 报告最终状态并提供完成工作摘要

注：此命令假设 `tasks.md` 中存在完整的任务分解。如果任务不完整或缺失，建议先运行 `/speckit-test.tasks` 重新生成任务列表。
