# 测试规格：注册登录

> **版本**: `001-register-login`
> **创建日期**: 2026-05-13
> **测试类型**: 功能测试 + 安全测试
> **输入**: 用户通过登录页（用户名+密码）或注册页（用户名+密码+邮箱）完成身份认证，认证通过后获得JWT Cookie令牌并重定向至控制台首页。

---

## 测试范围 *（必填）*

### 包含

- 用户注册新账户的完整流程（表单展示、输入校验、账户创建、自动登录、重定向）
- 用户登录的完整流程（表单展示、凭据验证、Cookie签发、重定向、"记住我"行为）
- 登录失败场景（错误凭据、锁定机制）
- 用户退出登录流程（Cookie清除、重定向、防后退绕过）
- API 认证接口（登录、Token续签、登出）的请求/响应/错误处理
- 安全机制行为验证（CSRF防护、XSS输入清洗、IP速率限制、登录失败锁定、Cookie安全属性、Token黑名单）
- 边界场景（Cookie过期、已认证用户访问登录/注册页的重定向）

### 不包含

- 单元测试
- 性能压测（如高并发下的极限吞吐量）
- 多因素认证（MFA）、第三方社交登录、短信验证码登录（不在 v1 范围内）
- 邮箱验证激活流程（不在 v1 范围内）

---

### 用户故事 5 - 通过 API 完成登录与登出（优先级：P2）

#### 前置条件

- 系统已启动，`/api/v1/auth/*` 端点可访问
- 系统中已存在至少一个有效用户账户
- 调用方可发送 HTTP 请求到指定端点

#### 验收场景

##### TC-040: API 登录-正确凭据

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 调用方发送 `POST /api/v1/auth/login` 请求，携带有效用户名和密码 | 请求体包含 username 和 password |
| 2 | 验证通过 | 返回 JWT Token（含用户ID、用户名、角色、签发时间、过期时间）和 refresh_token |
| 3 | 检查响应 | 不设置 HttpOnly Cookie，由调用方自主管理令牌存储 |
| 4 | 检查响应时间 | 请求在 2 秒内返回 |

##### TC-041: API 登录-错误凭据

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 发送 `POST /api/v1/auth/login` 请求，携带错误用户名或密码 | 请求体包含无效 username/password |
| 2 | 验证失败 | 返回 401 错误及提示"用户名或密码错误" |
| 3 | 检查响应格式 | 不区分用户名错误或密码错误 |

##### TC-042: API 登录-账号锁定

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 针对已存在用户名，连续5次发送错误凭据的请求 | 每次返回 401 |
| 2 | 第6次调用 `POST /api/v1/auth/login` | 返回 423 错误及"账号已被临时锁定，请 15 分钟后再试" |

##### TC-043: API Token 续签-有效 refresh_token

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 调用方持有有效的 JWT Token 和 refresh_token | Token 尚未过期 |
| 2 | 调用 `POST /api/v1/auth/refresh` 携带 valid refresh_token | 返回新的 JWT Token 和新 refresh_token |
| 3 | 尝试使用原 refresh_token | 原 refresh_token 失效，续签被拒 |

##### TC-044: API Token 续签-过期 refresh_token

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 调用方持有已过期 refresh_token | Token 已过期 |
| 2 | 调用 `POST /api/v1/auth/refresh` | 返回 401 错误及"刷新令牌已过期" |

##### TC-045: API 登出-Token黑名单化

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 调用方持有有效 JWT Token | 认证成功 |
| 2 | 调用 `POST /api/v1/auth/logout` 携带该 Token | Token 立即加入黑名单 |
| 3 | 使用已注销 Token 调用任意需要认证的 API | 返回 401 未授权 |

##### TC-046: 访问无认证的 API 端点

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 无有效认证凭据 | 请求不携带 Token |
| 2 | 调用任意需认证的 API 端点（如 `/api/v1/auth/refresh`、`/api/v1/auth/logout`） | 返回 401 未授权错误 |

##### TC-047: API IP速率限制

- **优先级**: P2

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 同一 IP 在 1 分钟内连续发送 10 次 `POST /api/v1/auth/login` 请求 | 所有请求正常处理 |
| 2 | 在同一分钟内发送第11次请求 | 返回 429 错误及"请求过于频繁，请稍后重试" |

## 边缘情况与异常测试 *（必填）*

### EC-001: JWT Cookie 过期-自动重定向

- **场景**: 已登录用户的JWT Cookie到期后访问受保护页面
- **严重程度**: 高

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 用户持有已过期的 JWT Cookie | Cookie 已失效 |
| 2 | 访问 `/console/dashboard` 或任意 `/console/*` 路由 | 自动重定向至 `/auth/login` |
| 3 | 查看登录页顶部 | 显示"会话已过期，请重新登录" |

### EC-002: CSRF Token 缺失/无效

- **场景**: 登录表单提交时未携带或携带无效的 CSRF Token
- **严重程度**: 高

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 访问 `/auth/login` 获取 CSRF Token | CSRF Token 被签发 |
| 2 | 提交登录请求时使用无效或缺失的 CSRF Token | 返回 403 拒绝登录 |

### EC-003: XSS/SQL注入输入清洗

- **场景**: 用户名输入框提交包含 HTML 标签或 SQL 注入字符
- **严重程度**: 高

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 在用户名字段输入包含 `<script>alert('xss')</script>` 或 SQL 注入字符的内容 | 输入提交 |
| 2 | 系统接收输入 | 服务端清洗非法字符，拒绝或清理后存储 |

### EC-004: 网络连接失败-注册行为

- **场景**: 用户点击"注册"时网络已断开
- **严重程度**: 中

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 用户填写完整注册表单 | 表单内容有效 |
| 2 | 在网络断开状态下点击"注册" | 显示"网络连接失败"提示 |

### EC-005: 统一 API 响应格式

- **场景**: 所有 API 端点返回统一 JSON 响应结构
- **严重程度**: 中

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 调用任意 API 端点（登录、续签、登出等） | 请求完成 |
| 2 | 检查响应体 | 包含成功/失败标识、数据负载、错误码与错误描述 |

### EC-006: 审计日志记录

- **场景**: 所有登录/登出操作记录审计日志
- **严重程度**: 中

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 执行成功登录 | 登录完成 |
| 2 | 检查审计日志 | 日志包含操作人（用户名）、时间戳、IP地址、操作结果（成功） |
| 3 | 执行错误凭据登录 | 登录失败 |
| 4 | 检查审计日志 | 日志包含操作人、时间戳、IP地址、操作结果（失败） |
