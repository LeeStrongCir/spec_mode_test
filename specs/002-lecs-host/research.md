# 研究记录: LECS主机管理功能

> **特性**: 002-lecs-host
> **创建日期**: 2026-05-12
> **目的**: 记录测试计划阶段的技术调研与决策，所有 `[NEEDS CLARIFICATION]` 已在此阶段解决。

## 技术决策

### 决策 1: 集成测试框架选型

- **选择了什么**: pytest + httpx（异步 Web 框架测试客户端）+ pytest-xdist（并行执行）
- **理由**: pytest 是 Python 生态中最成熟的测试框架，原生支持 fixture、参数化测试和插件生态。pytest-xdist 提供开箱即用的并行执行能力，满足宪章"测试框架必须支持并行执行"的强制要求。httpx 提供 AsyncClient 用于测试异步 Web API。
- **考虑的替代方案**: 
  - unittest: Python 内置但缺乏高级 fixture 和参数化能力
  - pytest + testcontainers: 更适合真实数据库集成测试，但本功能使用 SQLite 内存库即可满足隔离性需求，无需额外复杂度

### 决策 2: 端到端测试工具选型

- **选择了什么**: Playwright（TypeScript）
- **理由**: Playwright 提供自动等待、多浏览器支持、网络拦截和强大的定位器策略。宪章未指定工具，但 Playwright 的 `data-testid` 定位策略与"分层边界规则"中 E2E 测试的定位要求高度匹配。
- **考虑的替代方案**:
  - Cypress: 仅限 Chromium，不支持多浏览器
  - Selenium: 手动等待逻辑复杂，flaky test 风险高

### 决策 3: 测试数据隔离策略

- **选择了什么**: 每个测试使用独立 SQLite 内存数据库（`:memory:`），通过 `pytest.fixture(scope="function")` 保证作用域
- **理由**: SQLite 内存库启动速度极快（毫秒级），天然保证测试间无共享状态（符合宪章"测试之间无共享可变状态"要求）。每个 fixture 创建全新连接，避免状态泄漏。
- **考虑的替代方案**:
  - 共享测试数据库 + 事务回滚: 速度略快，但并发测试时可能产生锁冲突
  - Testcontainers（真实 PostgreSQL）: 更贴近生产但启动慢（秒级），对快速反馈循环不友好

### 决策 4: 异步任务队列 Mock 策略

- **选择了什么**: `unittest.mock.patch` mock Celery `delay()` / `apply_async()` 方法
- **理由**: 异步任务的执行可靠性由运维团队通过基础设施测试覆盖（见 spec.md 覆盖缺口 GAP-002）。本功能集成测试仅验证"任务提交逻辑正确性"——即业务逻辑是否正确调用了 `delay()`。Mock 方式消除了对外部消息队列（Redis/RabbitMQ）的依赖，使集成测试可以在无外部服务的 CI 环境中运行。
- **考虑的替代方案**:
  - Celery eager mode（`CELERY_TASK_ALWAYS_EAGER = True`）: 同步执行任务，更贴近真实但引入不必要的依赖和复杂度
  - 真实 Celery Worker: 需要完整消息队列基础设施，集成测试范围不应包含此复杂度

### 决策 5: 时间敏感逻辑测试策略

- **选择了什么**: `freezegun` 冻结/快进时间
- **理由**: 规格中定义了多个时间相关行为：创建超时 60 秒、状态轮询 3 秒间隔、关机/启动约 10 秒。`freezegun` 允许在测试中精确控制时间流动，使这些场景可以在毫秒级完成验证，而非等待真实时间流逝。
- **考虑的替代方案**:
  - 硬编码缩短超时阈值（如将 60 秒改为 1 秒用于测试）: 侵入生产逻辑，不推荐
  - 真实等待（`time.sleep(60)`）: 测试执行时间不可接受

### 决策 6: 计费估算测试方式

- **选择了什么**: Mock 计费服务返回固定费率表，测试验证前端公式正确性
- **理由**: 规格明确了计费公式（包年/包月 = 月费×月份，按需 = 月费÷30）。集成测试验证的是"公式实现正确性"而非"与外部计费服务的一致性"（后者在计费系统联调阶段覆盖，见 GAP-001）。
- **考虑的替代方案**:
  - 调用真实计费服务: 引入外部依赖，违背集成测试隔离原则
  - 硬编码费率到测试代码: 与 spec.md 规格表不一致时易产生漂移，使用固定 JSON fixture 更好

### 决策 7: 认证测试策略

- **选择了什么**: 真实 JWT 签发逻辑（使用固定测试 secret），mock 认证系统的外部验证
- **理由**: JWT 签发和验证逻辑是本功能认证链路的核心部分，必须真实执行以验证完整性。但认证系统的用户数据库、密码验证等不在测试范围内，通过创建测试用户 fixture 替代。
- **考虑的替代方案**:
  - 完全 mock JWT 逻辑: 无法验证 token 签发/解析的正确性
  - 调用真实认证服务: 引入不必要的跨服务依赖

## [NEEDS CLARIFICATION] 解决记录

本次规格中未遗留任何 `[NEEDS CLARIFICATION]` 标记。spec.md 在创建时已使用合理默认值完成所有决策：

| 原潜在不确定项 | 采用的默认值 | 依据 |
