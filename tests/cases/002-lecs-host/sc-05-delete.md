# SC-05: 安全主机删除测试用例

> **场景**: 验证运行态主机删除拦截、已关机态/创建失败态主机的异步软删除流程及配额释放
> **功能模块**: LECS 主机管理 — 安全删除
> **编写日期**: 2026-05-12
> **文档版本**: v1.0

---

## 前置条件

| # | 条件 |
|---|------|
| 1 | 测试人员已使用有效账号（含 LECS 主机管理权限）登录 Lee 云平台控制台 |
| 2 | 系统中已存在至少 3 台 LECS 主机，分别处于以下状态： |
| 2a | 主机 A：状态为"正常" (normal)，运行中 |
| 2b | 主机 B：状态为"已关机" (stopped)，已执行关机操作 |
| 2c | 主机 C：状态为"创建失败" (failed)，创建流程异常终止 |
| 3 | 测试人员持有数据库只读账号或具备通过管理后台执行 SQL 查询的权限 |
| 4 | 浏览器为桌面端浏览器，分辨率 ≥ 1280px |
| 5 | 当前用户配额未满（活跃主机数 < 100）|

## 数据准备

| # | 数据 | 用途 |
|---|------|------|
| 1 | 主机 A（normal 态）— 记录主机 ID、主机名、当前配额计数 | TC-040 |
| 2 | 主机 B（stopped 态）— 记录主机 ID、主机名、当前配额计数 | TC-041 |
| 3 | 主机 C（failed 态）— 记录主机 ID、主机名、当前配额计数 | TC-042 |
| 4 | 数据库访问凭证（用于 soft delete 验证） | TC-040 ~ TC-042 通用 |

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 访问 `/console/lecs-hosts/list`，确认三台主机分别显示对应状态 | 列表中可见"正常"、"已关机"、"创建失败"三种状态的主机 |
| 2 | 记录每台主机的 UUID（从 URL 页面源码、Network 请求或列表数据列获取） | 获得主机 A/B/C 的唯一 ID |
| 3 | 记录当前用户的配额计数（列表页总条数或配额管理页面显示的值） | 获得初始配额计数 N |

## TC-040: 验证运行态主机删除拦截

| 属性 | 值 |
|------|------|
| **优先级** | P2 |
| **类型** | 手工功能测试（前端 UI + API 错误拦截） |
| **前置条件** | 前置条件 1~5 及数据准备全部完成；主机 A 状态为"正常" (normal) |

### 测试步骤

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 访问 `/console/lecs-hosts/list`，等待页面加载完成 | 列表页正常渲染，数据表格显示所有活跃主机 |
| 2 | 在列表中定位到主机 A（状态显示为"正常"） | 该主机行存在，状态标签显示"正常" |
| 3 | 查看主机 A 所在行的操作列，检查"删除"按钮 | "删除"按钮 **置灰**（disabled），鼠标悬停时不可点击，无弹窗触发 |
| 4 | 同时检查"关机"和"启动"按钮 | "关机"按钮可点击，"启动"按钮置灰（符合 normal 态按钮矩阵） |
| 5 | （API 验证）打开浏览器开发者工具 → Network 面板 | Network 面板准备就绪 |
| 6 | （API 验证）通过 curl 或 API 调试工具，以当前用户身份向运行态主机发起删除请求：`DELETE /api/v1/lecs-hosts/{host_a_id}` | 请求正常发出 |
| 7 | 在 Network 面板或 API 响应中检查 HTTP 状态码 | 返回 **403 Forbidden** |
| 8 | 检查响应体（Response Body） | 响应体为 JSON 格式，包含以下字段：<br>• `success: false`<br>• `error_code`：权限/状态拒绝相关错误码<br>• `error_message`：包含"请先将主机关机，再执行删除操作"或语义等效的中文提示 |
| 9 | 返回前端列表页，刷新页面 | 主机 A 仍存在于列表中，状态不变 |
| 10 | 连接数据库，执行软删除前置验证 SQL：`SELECT id, hostname, status, deleted_at FROM lecs_hosts WHERE id = '<host_a_id>';` | 查询返回 1 行，`deleted_at` 列为 **NULL**，`status` 仍为"正常" |

### 验证点

| # | 验证项 | 通过标准 |
|---|--------|----------|
| V1 | 删除按钮 UI 状态 | normal 态主机"删除"按钮置灰，不可点击 |
| V2 | API 删除拦截 | DELETE 请求返回 HTTP 403 |
| V3 | 错误消息内容 | 响应体 `error_message` 包含"请先将主机关机，再执行删除操作" |
| V4 | 前端数据一致性 | 刷新后主机 A 仍出现在列表中 |
| V5 | 数据库软删除标记 | `deleted_at` 保持为 NULL，未被修改 |

### 通过/失败判定

- **通过**：全部验证点 V1~V5 通过
- **失败**：任一验证点不通过

## TC-041: 验证已关机态主机的异步删除流程

| 属性 | 值 |
|------|------|
| **优先级** | P2 |
| **类型** | 手工功能测试（前端 UI + 异步状态机 + 数据库软删除 + 配额计数） |
| **前置条件** | 前置条件 1~5 及数据准备全部完成；主机 B 状态为"已关机" (stopped) |

### 测试步骤

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 访问 `/console/lecs-hosts/list`，等待页面加载完成 | 列表页正常渲染 |
| 2 | 在列表中定位到主机 B（状态显示为"已关机"） | 该主机行存在，状态标签显示"已关机" |
| 3 | 查看主机 B 所在行的操作列，检查各按钮状态 | "删除"按钮 **可点击**；"启动"按钮可点击；"关机"按钮置灰（符合 stopped 态按钮矩阵） |
| 4 | 点击主机 B 的"删除"按钮 | 弹出二次确认对话框，对话框包含删除警告信息及"确定"/"取消"按钮 |
| 5 | （可选 — 取消分支）点击"取消"按钮 | 对话框关闭，主机 B 状态不变，仍为"已关机"，列表无变化 |
| 6 | 再次点击主机 B 的"删除"按钮 | 二次确认对话框再次弹出 |
| 7 | 点击对话框中的"确定"按钮，确认执行删除 | 对话框关闭 |
| 8 | 立即观察主机 B 行的状态标签 | 状态立即变为 **"删除中"**（deleting） |
| 9 | 同时观察主机 B 行的操作列（关机 / 启动 / 删除按钮） | 三个按钮 **全部置灰**，不可点击 |
| 10 | 打开浏览器开发者工具 → Network 面板，查看发出的 DELETE 请求 | 存在一条 `DELETE /api/v1/lecs-hosts/{host_b_id}` 请求，返回 **202 Accepted** |
| 11 | 等待 3~5 秒，持续观察主机 B 行 | 主机 B 行从列表中 **消失** |
| 12 | 确认主机 B 行消失后，检查列表页顶部的配额计数/总条数 | 配额计数较删除前 **减少 1**（从 N → N-1） |
| 13 | 刷新列表页（F5 或点击刷新按钮） | 主机 B 不再出现在列表中 |
| 14 | 连接数据库，执行软删除验证 SQL：`SELECT id, hostname, status, deleted_at FROM lecs_hosts WHERE id = '<host_b_id>';` | 查询返回 1 行，`deleted_at` 列为 **非 NULL**（包含具体的删除时间戳），主机数据未被物理删除 |
| 15 | 执行列表查询 SQL（模拟前端列表页查询逻辑）：`SELECT id, hostname, status FROM lecs_hosts WHERE user_id = '<当前用户ID>' AND deleted_at IS NULL ORDER BY created_at DESC;` | 结果集中 **不包含** 主机 B 的记录 |
| 16 | 执行配额计数 SQL：`SELECT COUNT(*) AS active_count FROM lecs_hosts WHERE user_id = '<当前用户ID>' AND deleted_at IS NULL;` | `active_count` 值 = 删除前的配额计数 **减 1**（即 N-1） |

### 验证点

| # | 验证项 | 通过标准 |
|---|--------|----------|
| V1 | 删除按钮可点击 | stopped 态主机"删除"按钮可正常点击 |
| V2 | 二次确认对话框 | 删除操作前弹出确认对话框，需用户主动确认 |
| V3 | 删除中状态 | 确认后主机状态立即变为"删除中" |
| V4 | 按钮防冲突 | "删除中"态下所有操作按钮均置灰 |
| V5 | API 响应状态 | DELETE 请求返回 202 Accepted |
| V6 | 行消失时效性 | 3~5 秒后主机行从列表中消失 |
| V7 | 配额计数释放 | 配额计数 -1 |
| V8 | 软删除标记 | 数据库中 `deleted_at` 非 NULL |
| V9 | 列表查询过滤 | `WHERE deleted_at IS NULL` 查询不返回已删除主机 |
| V10 | 数据完整性 | 主机数据未被物理删除，仅标记 `deleted_at` |

### 通过/失败判定

- **通过**：全部验证点 V1~V10 通过
- **失败**：任一验证点不通过

### 异常恢复（如 V6 未通过）

| 情况 | 处理 |
|------|------|
| 等待超过 30 秒主机行仍未消失 | 刷新列表页；若仍可见，记录为缺陷 — 删除异步任务执行失败 |
| 刷新后主机 B 重新出现且状态不变 | 记录为缺陷 — 后端未执行软删除或前端未正确同步 |

## SQL 验证汇总

以下 SQL 语句在 TC-040 ~ TC-042 执行过程中用于数据库级验证：

### SQL-01：删除前主机状态确认

```sql
SELECT id, hostname, status, deleted_at
FROM lecs_hosts
WHERE id IN ('<host_a_id>', '<host_b_id>', '<host_c_id>');
```

**预期**：三行记录，`deleted_at` 均为 NULL，`status` 分别为 normal、stopped、failed。

### SQL-02：软删除标记验证（删除后执行）

```sql
SELECT id, hostname, status, deleted_at
FROM lecs_hosts
WHERE id IN ('<host_b_id>', '<host_c_id>');
```

**预期**：两行记录，`deleted_at` 均 **非 NULL**，包含 `YYYY-MM-DD HH:MM:SS` 格式的时间戳。

### SQL-03：列表查询过滤验证（模拟前端列表页逻辑）

```sql
SELECT id, hostname, status
FROM lecs_hosts
WHERE user_id = '<当前用户ID>'
  AND deleted_at IS NULL
ORDER BY created_at DESC;
```

**预期**：结果集中仅包含未被删除的活跃主机（主机 A 仍在，主机 B/C 不在）。

### SQL-04：配额计数验证（对比删除前后）

```sql
SELECT COUNT(*) AS active_count
FROM lecs_hosts
WHERE user_id = '<当前用户ID>'
  AND deleted_at IS NULL;
```

**预期**：
- 删除前：active_count = N
- 删除主机 B 后（TC-041）：active_count = N - 1
- 删除主机 C 后（TC-042）：active_count = N - 2
