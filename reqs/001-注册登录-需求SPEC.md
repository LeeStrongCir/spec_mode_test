# 功能规格说明书：注册登录

**功能分支**: `001-register-login`  
**创建日期**: 2026-05-11  
**状态**: 草稿  
**输入**: 用户通过登录页（用户名+密码）或注册页（用户名+密码+邮箱）完成身份认证，认证通过后获得JWT Cookie令牌并重定向至控制台首页。

## 用户场景与测试 *(必填)*

### 用户故事 1 —— 注册新账户

未注册用户访问 `/auth/register` 页面，页面展示用户名、密码、确认密码及邮箱输入框。用户填写完整信息并点击"注册"按钮后，系统校验输入合法性（用户名唯一性、密码强度、邮箱格式、两次密码一致），校验通过后创建新账户并自动登录。用户随后被重定向至控制台首页（`/console`）。

**设定该优先级的原因**：注册是用户进入平台的第一途径。与登录功能配合形成完整的认证生命周期入口。

**独立可测**：可通过新用户名+密码+邮箱组合提交注册，验证账户创建、自动登录、Cookie签发及重定向。同时验证各种校验拦截场景（重复用户名、弱密码、邮箱格式错误等）。

**验收场景**：

1. **前提** 未注册用户访问 `/auth/register` 页面，**操作** 用户查看页面，**预期** 页面展示用户名、密码、确认密码、邮箱输入框及"注册"按钮，底部提供"已有账户？返回登录"链接。
2. **前提** 输入框为空或信息不完整，**操作** 用户点击"注册"，**预期** 各未填字段下方显示对应的错误提示（如"请输入用户名"）。
3. **前提** 用户名已被其他用户占用，**操作** 用户填写并提交相同的用户名，**预期** 用户名下方显示"该用户名已被注册"的错误提示。
4. **前提** 密码与确认密码不一致，**操作** 用户点击"注册"，**预期** 确认密码框下方显示"两次输入的密码不一致"的错误提示。
5. **前提** 填写合法的用户名、密码（8-32位，含字母+数字+特殊字符）、确认密码及有效的邮箱地址，**操作** 用户点击"注册"，**预期** 按钮进入加载状态，账户创建成功后自动签发JWT Cookie并重定向至 `/console`。
6. **前提** 用户在注册页底部点击"已有账户？返回登录"，**操作** 点击该链接，**预期** 页面跳转至 `/auth/login`。

### 用户故事 2 —— 输入凭据完成登录

用户访问Lee云平台登录页（`/auth/login`），页面展示用户名输入框、密码输入框及"登录"按钮。用户输入正确的用户名和密码后点击"登录"按钮，系统验证凭据成功后签发JWT Cookie，将用户重定向至控制台首页（`/console`）。

**设定该优先级的原因**：这是用户访问平台的日常入口。整个系统的所有功能都依赖于用户成功通过认证。

**独立可测**：可通过输入正确的用户名和密码组合，验证登录成功、Cookie签发、重定向行为来进行完整测试。

**验收场景**：

1. **前提** 用户访问 `/auth/login` 页面，**操作** 用户查看页面，**预期** 页面展示用户名输入框、密码输入框、"记住我"复选框及"登录"按钮。
2. **前提** 输入框为空，**操作** 用户不填任何内容直接点击"登录"，**预期** 用户名和密码输入框下方分别显示"请输入用户名"和"请输入密码"的错误提示。
3. **前提** 输入正确的用户名和密码，**操作** 用户点击"登录"，**预期** 按钮进入加载状态（防止重复提交），2秒内完成认证，页面重定向至 `/console`，且浏览器Cookie中包含有效的JWT Token。
4. **前提** 用户勾选"记住我"后登录，**操作** 关闭浏览器后重新访问平台，**预期** 用户无需重新输入凭据，自动通过Cookie认证进入控制台。

### 用户故事 3 —— 处理登录失败

用户在登录页输入错误的用户名或密码。系统验证凭据失败后，在页面顶部显示通用错误提示，但不明确指出"用户名错误"还是"密码错误"，以防止用户名枚举攻击。失败后用户名输入框保留已填值，密码框清空。

**设定该优先级的原因**：登录失败是最常见的用户路径。错误提示需要在用户友好性和安全性之间保持平衡。

**独立可测**：可通过输入错误凭据验证错误提示显示、密码框清空、用户名框保留的行为。

**验收场景**：

1. **前提** 用户输入不存在的用户名和任意密码，**操作** 点击"登录"，**预期** 页面顶部显示"用户名或密码错误"，用户名输入框保留已填内容，密码输入框被清空。
2. **前提** 用户输入存在的用户名但错误的密码，**操作** 点击"登录"，**预期** 页面顶部显示"用户名或密码错误"，用户名输入框保留已填内容，密码输入框被清空。
3. **前提** 连续5次登录失败（同一用户名），**操作** 第6次尝试登录，**预期** 该用户名被锁定15分钟，页面提示"账号已被临时锁定，请 15 分钟后再试"。

### 用户故事 4 —— 退出登录

用户已在控制台中，点击顶部导航栏用户头像菜单中的"退出登录"选项。系统清除JWT Cookie，将用户重定向至登录页（`/auth/login`），并在登录页顶部显示"您已成功退出登录"的成功提示。

**设定该优先级的原因**：安全退出是认证生命周期的重要一环，确保会话不被后续使用者接管。

**独立可测**：可通过已登录状态下执行退出操作，验证Cookie清除、重定向及提示消息。

**验收场景**：

1. **前提** 用户已登录并处于控制台任一页面，**操作** 点击右上角用户头像菜单中的"退出登录"，**预期** JWT Cookie被清除，页面重定向至 `/auth/login`，顶部显示"您已成功退出登录"。
2. **前提** 用户已被重定向至登录页，**操作** 点击浏览器的"后退"按钮，**预期** 用户停留在登录页，无法通过浏览器缓存返回到控制台页面。

### 用户故事 5 —— 通过 API 完成登录与登出（面向其他服务/平台）

外部服务或平台需要以编程方式对用户身份进行认证和会话管理。系统必须提供一组 RESTful API 接口，覆盖登录验证、Token 续签和登出注销操作，供自动化流程或其他服务调用。注册操作不提供 API 接口——新账户创建仅支持通过网页完成。

**设定该优先级的原因**：认证能力不应仅限于控制台手动操作。自动化运维流程、第三方平台集成、内部服务编排都需要通过 API 复用相同的身份验证和会话管理能力——缺少 API 将迫使外部系统进行重复开发或绕过规范流程直接使用底层凭据，引入安全与一致性风险。不提供注册 API 可防止批量恶意注册和滥用。

**独立可测**：可通过直接调用 API 端点，验证认证凭证校验、JWT Token 返回格式、续签逻辑、登出 Token 失效及并发约束。

**验收场景**：

1. **前提** 调用方持有有效的用户名和密码凭据，**操作** 调用 `POST /api/v1/auth/login` 提交登录请求，**预期** 验证通过后返回 JWT Token（含用户ID、用户名、角色、签发时间、过期时间）及刷新令牌（refresh_token），**不**设置 HttpOnly Cookie——调用方可自主管理令牌存储。
2. **前提** 调用方输入错误的用户名或密码，**操作** 调用 `POST /api/v1/auth/login`，**预期** 拒绝请求，返回 401 错误及通用提示"用户名或密码错误"，不区分具体错误类型。
3. **前提** 同一用户名连续5次登录失败，**操作** 第6次调用 `POST /api/v1/auth/login`，**预期** 拒绝请求，返回 423 错误及"账号已被临时锁定，请 15 分钟后再试"。
4. **前提** 调用方的 JWT Token 尚未过期但剩余有效期不足50%，**操作** 调用 `POST /api/v1/auth/refresh` 并携带 refresh_token，**预期** 返回新的 JWT Token 和刷新令牌，原刷新令牌失效。
5. **前提** 调用方的 JWT Token 已过期，**操作** 调用 `POST /api/v1/auth/refresh` 并携带 refresh_token，**预期** 拒绝请求，返回 401 错误及"刷新令牌已过期"。
6. **前提** 调用方持有有效的 JWT Token，**操作** 调用 `POST /api/v1/auth/logout`，**预期** Token 立即加入黑名单并拒绝后续访问，返回注销成功响应。
7. **前提** 调用方无有效认证凭据，**操作** 调用任意认证 API 端点（除登录外），**预期** 返回 401 未授权错误。
8. **前提** 同一 IP 在 1 分钟内发起超过 10 次登录 API 请求，**操作** 第11次调用 `POST /api/v1/auth/login`，**预期** 拒绝请求，返回 429 错误及"请求过于频繁，请稍后重试"。

### 边界场景与异常处理

- **Cookie过期**：JWT Cookie到期后，用户访问受保护页面（`/console/*`）时自动重定向至 `/auth/login`，并在登录页顶部显示"会话已过期，请重新登录"。
- **CSRF防护**：登录表单提交时携带CSRF Token，防止跨站请求伪造攻击。Token不匹配时拒绝登录并返回403。
- **并发锁定**：同一IP在1分钟内发起超过10次登录请求，则触发IP限速，后续请求在锁定窗口内返回429 Too Many Requests，提示"请求过于频繁，请稍后重试"。
- **网络断连**：网络断连期间点击"登录"，显示"网络连接失败"提示；网络恢复后用户可重试登录，表单已填值保留。
- **XSS防护**：用户名输入框拒绝包含HTML标签、JavaScript代码、SQL注入字符的特殊输入，提交时服务端清洗非法字符。
- **密码强度**：创建新用户时，密码支持英文大小写字母、数字、特殊字符组合，长度8-32个字符。不满足时在输入框下方显示错误提示。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统 **必须** 在 `/auth/login` 页面提供用户名输入框和密码输入框供用户输入凭据。
- **FR-002**：系统 **必须** 对用户提交的用户名和密码进行验证，验证通过后签发HttpOnly的JWT Cookie。
- **FR-003**：系统 **必须** 在用户登录成功后将其重定向至控制台首页（`/console`）或用户之前试图访问的受保护页面。
- **FR-004**：系统 **必须** 在用户名或密码为空时于表单提交前拦截，分别在对应输入框下方展示"请输入用户名"和"请输入密码"。
- **FR-005**：系统 **必须** 在登录失败时仅展示通用错误提示"用户名或密码错误"，不区分用户名错误或密码错误。
- **FR-006**：系统 **必须** 对同一用户名连续5次登录失败后锁定该账号15分钟，期间拒绝该用户名的登录尝试。
- **FR-007**：系统 **必须** 在用户成功退出登录后清除JWT Cookie并重定向至 `/auth/login`，于登录页展示"您已成功退出登录"。
- **FR-008**：系统 **必须** 对用户勾选"记住我"的会话签发有效期为30天的JWT Cookie，未勾选的签发有效期为24小时的Cookie。
- **FR-009**：系统 **必须** 在所有受保护路由（`/console/*`）上执行JWT Cookie校验，未持有有效Cookie的请求重定向至 `/auth/login`。
- **FR-010**：系统 **必须** 对所有登录请求强制执行CSRF Token校验，缺少或无效CSRF Token的请求返回403。
- **FR-011**：系统 **必须** 对同一IP地址实施登录请求速率限制——1分钟内最多10次登录请求，超出返回429。
- **FR-012**：系统 **必须** 禁止 `/auth/login` 页面在用户已持有有效JWT Cookie时展示——已认证用户访问登录页时，自动重定向至 `/console`。
- **FR-013**：系统 **必须** 对所有登录/登出操作记录审计日志，含操作人（用户名）、时间戳、IP地址、操作结果（成功/失败/锁定）。
- **FR-014**：系统 **必须** 在 `/auth/register` 页面提供用户名、密码、确认密码、邮箱输入框供用户注册新账户。
- **FR-015**：系统 **必须** 在用户提交注册时校验用户名唯一性，若用户名已被占用则拒绝注册并提示"该用户名已被注册"。
- **FR-016**：系统 **必须** 在用户提交注册时校验密码强度（8-32个字符，含英文字母、数字、特殊字符），不满足时提示错误。
- **FR-017**：系统 **必须** 在用户提交注册时校验密码与确认密码一致性，不一致时提示"两次输入的密码不一致"。
- **FR-018**：系统 **必须** 在用户提交注册时校验邮箱格式合法性，邮箱格式不合法时提示错误。
- **FR-019**：系统 **必须** 在注册成功后自动签发JWT Cookie并将用户重定向至 `/console`。
- **FR-020**：系统 **必须** 对所有注册请求执行CSRF Token校验，与登录请求共享防护机制。
- **FR-021**：系统 **必须** 禁止已持有有效JWT Cookie的用户访问注册页——已认证用户访问 `/auth/register` 时，自动重定向至 `/console`。
- **FR-022**：系统 **必须** 提供 `POST /api/v1/auth/login` 接口，供外部服务提交用户名和密码进行认证。请求体包含用户名和密码，验证通过后返回 JWT Token（含用户ID、用户名、角色、签发时间、过期时间）和刷新令牌（refresh_token）。
- **FR-023**：系统 **必须** 对 `POST /api/v1/auth/login` 接口的认证凭据进行校验，错误凭据返回 401 错误及通用提示"用户名或密码错误"，不区分错误类型。
- **FR-024**：系统 **必须** 对 `POST /api/v1/auth/login` 接口执行与网页登录一致的失败锁定策略——同一用户名连续5次失败后锁定15分钟，返回 423 错误。
- **FR-025**：系统 **必须** 对 `POST /api/v1/auth/login` 接口执行与网页登录一致的IP速率限制——1分钟内最多10次登录请求，超出返回 429 错误。
- **FR-026**：系统 **必须** 提供 `POST /api/v1/auth/refresh` 接口，供外部服务刷新JWT Token。请求携带有效 refresh_token 时返回新 JWT Token 和新 refresh_token，原 refresh_token 失效。
- **FR-027**：系统 **必须** 对 `POST /api/v1/auth/refresh` 接口校验 refresh_token 有效性，过期或无效的 refresh_token 返回 401 错误。
- **FR-028**：系统 **必须** 提供 `POST /api/v1/auth/logout` 接口，供外部服务注销JWT Token。调用时 Token 立即加入黑名单，拒绝后续访问。
- **FR-029**：系统 **必须** 为除登录外的所有 API 接口强制执行认证鉴权，未持有有效 JWT Token 的请求返回 401 错误。
- **FR-030**：系统 **必须** 为 API 接口提供统一 JSON 响应格式（含成功/失败标识、数据负载、错误码与错误描述），确保调用方一致解析。

### 关键实体

- **用户帐户**：平台注册用户。属性包括用户名（唯一）、密码（BCrypt加密哈希）、邮箱（唯一）、帐户状态（活跃/锁定）、最后登录时间及IP、登录失败计数及最近失败时间、注册时间戳。
- **JWT Cookie**：包含用户ID、用户名、角色（admin/user）及签发/过期时间的JSON Web Token。存储方式为HttpOnly、Secure、SameSite=Strict。
- **CSRF Token**：用于防止跨站请求伪造的一次性令牌，绑定到用户当前会话，随表单提交时提交。
- **JWT Token**：API 认证使用的令牌。属性包括用户ID、用户名、角色（admin/user）、签发时间、过期时间。API 登录不设置 HttpOnly——调用方可自主存储。
- **刷新令牌（Refresh Token）**：用于续签 JWT Token 的独立令牌。属性包括签发时间、过期时间、一次性使用标志（单次使用后立即失效）。
- **Token 黑名单**：存储已注销或已撤销的 JWT Token 标识。API 登出时 Token 加入黑名单，后续访问被拒绝。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：用户在输入正确凭据后2秒内完成登录并重定向至控制台首页。
- **SC-002**：错误凭据登录拦截提示准确率为100%，不泄露用户名存在的任何信息。
- **SC-003**：同一用户名连续失败5次后账号锁定成功率为100%，锁定15分钟内无法登录。
- **SC-004**：JWT Cookie签发成功率和HttpOnly设置覆盖率100%——XSS攻击无法窃取令牌。
- **SC-005**：同一IP每分钟10次登录限速触发率100%——防止暴力破解和DDoS。
- **SC-006**：未认证用户访问 `/console/*` 页面时100%重定向至 `/auth/login`。
- **SC-007**：用户名唯一性校验成功率100%——重复用户名注册被准确拦截并正确提示。
- **SC-008**：注册成功后自动登录成功率100%——用户无需再次输入凭据即完成注册后进入控制台。
- **SC-009**：API 登录接口 2 秒内返回 JWT Token 和 refresh_token，成功率 100%。
- **SC-010**：API 失败凭据拦截准确率 100%——错误凭据返回 401，不泄露用户名存在信息。
- **SC-011**：API Token 续签成功率 100%——有效 refresh_token 在任意时间续签，过期 token 续签被拒率 100%。
- **SC-012**：API 登出后 Token 黑名单拦截率 100%——已注销 Token 无法通过任何 API 鉴权。

## 假设

- 复用现有的JWT签发和验证基础设施，不做修改。
- 移动端和平板端的响应式设计在v1范围内需要支持（登录页适配320px及以上视口）。
- 登录失败锁定、IP限速等安全策略通过Redis/内存存储临时状态。
- "记住我"功能依赖Cookie持久化，用户可手动清除浏览器Cookie以登出。
- 审计日志以异步方式写入，不影响登录请求的响应延迟。
- v1版本仅支持用户名+密码认证；多因素认证（MFA）、第三方社交登录、短信验证码登录不在v1范围内。
- 注册成功后无需邮箱验证激活——账户即刻可用；邮箱验证功能不在v1范围内。
- API 接口版本采用 `/api/v1/` 前缀，后续不兼容变更通过版本号升级，不影响现有调用方。
- 注册操作不提供 API 接口——新账户创建仅能通过控制台网页完成，防止批量恶意注册。
- 外部服务调用 API 时，通过返回的 JWT Token 进行认证，不再依赖浏览器的 HttpOnly Cookie。
- Token 黑名单存储通过 Redis/内存实现，保证高性能查检；黑名单条目随原 Token 过期自动清理。
